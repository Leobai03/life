# Life SOP

## Core Idea

Use this skill to help a user build a practical personal life SOP: a lightweight operating system for living, working, resting, relating, and recovering without sacrificing health for short-term results.

Do not position the SOP as a perfect life plan. Position it as a versioned system that serves the user's current stage. The output should make the user's next 7-30 days easier to execute.

## Operating Principles

- Protect the user's body and baseline stability before optimizing ambition.
- Treat disorder, exhaustion, avoidance, and self-hate as signals to diagnose, not moral failures.
- Reduce the user's cognitive load. Give fewer, clearer actions.
- Prefer "minimum viable rhythm" over dramatic transformation.
- Do not prescribe one universal ideal day. Build around the user's real constraints, energy, and priorities.
- Include emotional, relational, and bodily life, not only career and productivity.
- Make every SOP version runnable, reviewable, and adjustable.

## Workflow

### 1. Stabilize the Frame

Start by reflecting the user's current stage in plain language. Name the pattern without shaming them.

Use this structure:

```text
You are not simply [negative label]. You are currently in a pattern of [system diagnosis].
The first goal is not to become perfect. The first goal is to restore order and momentum.
```

Examples of system diagnoses:

- External pressure dropped, so rhythm collapsed.
- Too many opportunities created decision fatigue.
- Sleep and food are unstable, so execution feels harder than it should.
- Online feedback is hijacking attention and self-worth.
- Emotional or relational needs are leaking into work time.

### 2. Gather Inputs

Ask for only the missing information needed to design v1. If the user has already provided enough context, proceed directly.

Use [question-bank.md](references/question-bank.md) when you need deeper prompts.

Minimum input fields:

- Current stage
- Top 1-3 things to protect
- Top 1-3 desired results
- Biggest disorder points
- Most unstable time window
- Current sleep, food, movement
- Most effective time window, if any
- Life state the user most wants to stop repeating

### 3. Define The Stage Theme

Summarize the user's next-stage theme in one sentence.

Good format:

```text
[Protect X], [restore Y], and [build Z] without [main cost to avoid].
```

Examples:

- Stabilize cash flow, restore rhythm, and rebuild creative output without burning out.
- Protect health, reduce chaos, and find a real direction before chasing scale.
- Keep work moving while rebuilding sleep, food, movement, and emotional steadiness.

### 4. Choose 3 Priorities

Limit v1 to 3 priorities. If the user gives fewer, do not force a third. If they give too many, merge or defer.

Priority categories:

- Body: sleep, food, movement, medical basics, energy
- Work: business, job, income, execution
- Creation: writing, content, public output, portfolio
- Learning: reading, skill building, technical foundation
- Emotion: anxiety, avoidance, self-worth, loneliness
- Relationships: family, friends, intimacy, collaborators
- Money: cash flow, budgeting, risk
- Environment: room, desk, city, offline life

### 5. Build The SOP

Use [sop-template.md](references/sop-template.md) as the output structure.

The SOP should include:

- Stage theme
- Protected priorities
- Daily pass line
- Morning/starting protocol
- Main work block
- Content or learning trace
- Body baseline
- Evening shutdown protocol
- Emotional emergency protocol
- Weekly review
- Boundaries

Keep the daily pass line small enough that a low-energy user can still succeed.

### 6. Make A Short Execution Checklist

If the user asks for execution help, generate a 7-day, 14-day, or 30-day checklist.

Default choice:

- 7 days for users in acute chaos or very low energy.
- 14 days for users who can run a reset but need support.
- 30 days for users who already have stable sleep and baseline execution.

Use [review-template.md](references/review-template.md) for daily check-ins and review prompts.

### 7. Iterate Instead Of Concluding

End with a clear review cadence.

Recommended cadence:

- Daily: 3-6 line check-in.
- Day 7: first review and reduction.
- Day 14: second review and v2 design.
- Day 30: decide what becomes permanent.

Never imply that v1 is the user's final life answer.

## Output Style

- Be warm, direct, and concrete.
- Avoid motivational performance language.
- Avoid making the user feel behind.
- Use simple checklists and short paragraphs.
- Name the user's real tradeoffs without drama.
- Replace self-attacking labels with operational diagnoses.

## Safety And Scope

This skill is not therapy, medical diagnosis, or legal/financial advice. If the user describes self-harm risk, severe depression, eating disorder symptoms, substance dependence, abuse, or medical danger, prioritize immediate safety and encourage professional or emergency support.

For ordinary disorder, low energy, anxiety, loneliness, or avoidance, stay practical and supportive.
